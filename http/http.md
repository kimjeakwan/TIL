# http
- 텍스트 기반의 통신 규약으로 인터넷에서 데이터를 주고받을 수 있는 프로토콜이다.
- 포로토콜은 컴퓨터 내부에서  또는 컴퓨터 사이에서 데이터의 교환 방식을 정의하는 규칙 체계 라 할수있다.
# http 의 특징
1. 클라이언트-서버 구조
- http 의 특징 인 클라이언트-서버 구조는 클라이언트 즉  웹 브라이저,핸드폰,노트북 등 이 서버에 요청을 하면 서버는 응답을 보내는 구조이다.
2. 무상태(stateless) 프로토콜
- 서버가 클라이언트이의 상태를 보존하지 않는 특징이다.
- 그 대신 클라이언트 측에서 해당 정보를 기억하며 매 요청마다 이전 상태까지 모두 포함해서 보낸다.
- 단점은 클라이언트가 이전 요청까지 붙여야하는 단점이 있지만 장점으로는 서버 에 부담하는 관리비가 줄고 클라이언트가 모든 데이터를 담고 요청하기 떄문에 아무 서버나 쓸수 있다.
- 상태 유지(stateful) 은 서버가 클라이언트이 상태를 보존한다. 클라이언트의 요청1에 대한 상태를 해당 요청을 받은 서버A가 기억하고 있는 것이다. 따라서 항상 같은 서버가 유지되어야 한다.
3. 비연결성(connectionless)
- 요청이 있을떄만 연결을 유지하고 이후 응답으로 데이터를 보낸 후 연결을 종료 하는 것이다.
- 데이터 전송이 필요할 떄 만 연결을 하기 때문에 동시에 처리 하는 요청이 줄어든다.
- TCP/IP 연결을 새로 맺어야하므로 3 way handshake 시간이 추가된다.
- 3 way handshake는 통신이 가능한지 syn 을 보내고 통신이 가능하면 syn-ack을 받는다. 이떄 문제가 없으면 통신하고자 하는 메세지인 ACK 보내고 양방향 통신이 되는 과정 이다.
- 지속연결이란 지속연결은 서버가 클라이언트의 요청의 대해서 응답한 후 TCP연결을 유지하는 방식으로 같은 클라이언트-서버간의 요청-응답이 동일한 TCP연결을 통해 이루어지는 연결 방식 이다. 
- 한개의 TCP연결은 하나의 요청메시지와 하나의 응답메시지만 전송가능 하기 떄문에 서버의 동시접속자가 10만명이라면 서버는 10만개의 쓰레드가 필효하다.
# http 메서드
1. GET
- 리소스를 조회하는 메서드 이다.(리소스:사용할 수 있는 어떤 것 )
- 멱등성 이라는 개념을 지니고 있어, 여러 번 조회 요청을 하여도 리소는 변하지 않는다.
- 쿼리스트링을 통해서 전달한다.(쿼리스트링은 URL의 뒤에 입력 데이터를 함께 제공하는 가장 단순한 데이터 전달 방법이다 )
2. POST
- 주로 새로운 리소스를 생성 하는데 사용한다.
- 데이터를 메세지 바디에 쿼리 파라미터 형식으로 전달 한다.(쿼리 파라미터는 key-value 형식으로 되어 있다.)
- 데이터가 외부로 노출되지 않으므로 보안성이 좋다.
- Body에 담아 전송하므로, 메세지 길이의 제한이 없다.
3. put: 리소스를 완전히 대체한다.
- 예를 들어 A,B 라는 데이터가 존재하는데 C라는 데이터를 담아 put 요청을 보낸다면 A,B 데이터가 삭제되고 C 데이터로 대체한다.
4. PATCH
- 리소스를 부분 변경한다.
- 예를 들어 A,B 라는 데이터가 있는데 B=C 로 PATCH 요청을 보내면 B는 삭제하고 A,C  데이터 가 남는다. 
5. DELETE 
- 리소스를 제거하는 역할이다.
# http request 구조
1. start Line
- GET/search http/1.1
- 첫 번쨰는 http 의 메서드 해당 request가 의도한 action을 정의하는 부분.
- 2 번쨰는 해당 request가 전송되는 목표 uri이다 예를 들어 /login 이 있다.
- 3번째는 http 의 버젼이 있다.
2. headers 
- host는 요청이 존송되는 목표의 host url 이다.예를 들어 google.com
- User-Agent는 요청을 보내는 클라이언트의 대한 정보: 예를 들어, 웹브라우저에 대한 정보.
- accept는 서버에서 해당 타입에 데이터를 보내달라고 하는 요청
- connection는 해당 요청이 끝난후에 클라이언트와 서버가 계속해서 네트워크 컨넥션을 유지 할것인지 아니면 끊을것인지에 대해 지시하는 부분.
- Content-Type는 해당 요청이 보내는 메세지 body의 타입. 
- Content-Length는 메세지 body의 길이
3. body 는 해당 rqeust의 실제 내용이다.
```
POST /payment-sync HTTP/1.1

Accept: application/json
Accept-Encoding: gzip, deflate
Connection: keep-alive
Content-Length: 83
Content-Type: application/json
Host: intropython.com
User-Agent: HTTPie/0.9.3

{
    "imp_uid": "imp_1234567890",
    "merchant_uid": "order_id_8237352",
    "status": "paid"
}
```
- 여기 마지막 부분이 해당 rqeust의 내용이다.즉 바디 다.
# 웹 브라우저 요청 흐름 
1. 웹 브라우저에 URL 입력
2. IP는 DNS를 조회해서 얻고, PORT정보로 HTTP 요청 메세지를 생성한다.
3. socket 라이브러리를 통해 
헤더 부분에 IP,PORT 정보를 넣어 3 way-handshake로 클라이언트와 서버를 연결한다. 그 후 연결이 성공하면 http의 메세지를 TCP/IP 계층에 전달
4. 해더 부분에 출발지 IP,port,목적지 Ip,port 정보를 담고 데이터 부분에서 http 요청 메세지를 담아 tcp/ip  패킷을 생성한다. 
5. Tcp/ip 패킷이 클라이언트에서  웹 서버에 패킷을 전달한다.
6. 서버에 도착한 헤더 부분은 버리고 http 요청 메세지를 서버가 해석하여 http응답 메세지를 만든다.
7. 응답 패킷을 클라이언트에 전달한다. 응답 패킷이 클라이언트이 도착 하면 응답 패킷에 헤더 부분을 버리고 http  응답 메세지의 데이터를 화면에 보여준다.
- socket 은 프로세스가 드넓은 네트워크 세계로 데이터를 내보내거나 혹은 그 세계로부터 데이터를 받기 위한 실제적인 창구 이다.
